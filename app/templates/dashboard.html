Key concepts explained

🚫 Why accuracy fails here

  • Only 0.13% of transactions are fraud
  • Predicting "all legit" gives 99.87% accuracy
  • But catches zero fraud — completely useless
  • Use Precision, Recall, F1, and PR-AUC instead

🔬 Class imbalance handling

  • class_weight='balanced' — loss penalises fraud misses ~770× more
  • SMOTE — creates synthetic fraud via KNN interpolation
  • Both prevent model ignoring the rare class
  • We compared both — results in the table above

⚖️ Precision vs Recall

  • High recall → catch all fraud → more false alarms
  • High precision → fewer alarms → miss some fraud
  • Banks prefer recall (financial loss > customer friction)
  • Threshold tuning is how you control this trade-off

🌲 Why Random Forest wins

  • Learns non-linear patterns (e.g. TRANSFER + balance_zero = fraud)
  • 200 trees — more robust, less overfitting
  • Handles extreme outlier amounts naturally
  • Feature importance is interpretable for professors
About this dataset

📦 PaySim — Mobile Money Fraud Simulation

This is not the ULB credit card dataset. This is PaySim — a simulation of mobile money transactions based on a real financial company's logs from Africa. It was created for research and has real fraud patterns preserved.

Key difference from ULB: the features are interpretable (you can explain oldbalanceOrg, TRANSFER type, etc. to anyone) whereas ULB used PCA-anonymized V1–V28 columns. PaySim is actually a better dataset for learning because you can reason about why each feature matters for fraud.
